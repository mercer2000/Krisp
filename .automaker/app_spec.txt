  <project_name>Krisp — Open Brain Productivity Hub</project_name>

  <overview>
    Krisp is a comprehensive, database-backed productivity and AI knowledge hub that combines Kanban task management, multi-source email aggregation, meeting intelligence (via Krisp webhook transcripts), and an &quot;Open Brain&quot; persistent memory system — all unified under a single Next.js application with a Neon PostgreSQL database.

The project solves the &quot;memory problem&quot; in AI workflows: every time you open a new chat window or switch AI tools, your accumulated context resets to zero. Krisp addresses this by providing (1) a centralized productivity dashboard for managing tasks, emails, and meeting transcripts with semantic search, and (2) an Open Brain architecture — a database-backed, MCP-connected knowledge system where any AI tool (Claude, ChatGPT, Cursor, etc.) can query your accumulated context through a single open protocol.

Key goals:
- Eliminate context silos across AI tools by providing persistent, semantically searchable memory accessible via MCP
- Aggregate meetings, emails, and tasks into a single productivity surface with AI-powered search
- Enable sub-second thought capture (via Slack or any MCP client) with automatic embedding, classification, and metadata extraction
- Provide compounding knowledge advantage: every captured thought makes every future AI interaction richer
- Maintain full user ownership of data — no SaaS middlemen, single Postgres database, ~$0.10–$0.30/month operational cost
- Support the agent web: make all knowledge queryable by meaning (vector embeddings), not just keywords, through standard protocols any autonomous agent can speak
  </overview>

      <name>Implement application-layer encryption for sensitive user data</name>
      <description>Application-Layer Encryption for Sensitive User Data

Implement transparent, application-layer encryption for all database columns that contain sensitive user-owned content, ensuring that Krisp developers and direct database access cannot read plaintext data while the application continues to function normally for authenticated users.

---

Scope &amp; Affected Tables

Identify and encrypt all columns containing sensitive user-authored or user-received content across the following surfaces (audit other tables during implementation for any overlooked PII):

- `emails` table — subject, body, sender name, recipient fields, snippet
- `meeting_transcripts` table — full transcript text, summary, key points
- `meetings` table — title, description, participant names/emails
- `action_items` table — title, description, notes
- `kanban_cards` table — title, description, markdown body
- `brain_thoughts` / Open Brain memory table — thought content, metadata, tags
- `journal_entries` (if present) — all user-authored text fields
- Any other table containing user-entered free text, names, or contact information

---

Requirements

Encryption Strategy
- Implement AES-256-GCM symmetric encryption at the application layer, before data is written to Neon PostgreSQL, so the database stores only ciphertext
- Use a single master encryption key (MEK) stored in a secrets manager (environment variable as minimum viable approach; document upgrade path to AWS KMS, HashiCorp Vault, or Doppler)
- Generate a unique initialization vector (IV) per field per write; store IV alongside ciphertext (e.g., as a prefixed, base64-encoded string in the same column or a companion column)
- Do not encrypt columns used for filtering, sorting, or indexing (e.g., foreign keys, timestamps, status enums, boolean flags) — encrypt only free-text and PII payload columns

Key Management
- Define a `ENCRYPTION_KEY` environment variable (32-byte hex or base64) required at application startup; fail fast with a clear error if absent
- Document key rotation procedure: introduce a `key_version` column or prefix on each encrypted field, write a migration script that re-encrypts all rows with the new key, and keep the old key available during the rotation window
- Never log, expose via API response, or include in error traces the raw plaintext of encrypted fields

Encryption Utility Module
- Create a shared utility (e.g., `lib/encryption.ts`) that exports:
  - `encrypt(plaintext: string): string` — returns a self-contained ciphertext string (IV + tag + ciphertext, base64-encoded)
  - `decrypt(ciphertext: string): string` — reverses the above; returns plaintext
  - `encryptNullable(value: string | null): string | null` — handles nullable columns safely
  - `decryptNullable(value: string | null): string | null`
- All encryption/decryption must occur in this single module; no ad-hoc crypto elsewhere in the codebase

Drizzle ORM Integration
- Apply encryption transparently at the data access layer rather than in individual route handlers or components
- Introduce typed Drizzle custom column types or repository wrapper functions (`insertEncrypted`, `selectDecrypted`) so callers work with plaintext and encryption is invisible
- Ensure Drizzle migrations do not alter column types for already-stored ciphertext (columns remain `text`); add a migration comment documenting that these columns store encrypted values

Vector Embeddings &amp; Semantic Search
- Embeddings (pgvector columns) are generated from plaintext before encryption; the embedding itself is not sensitive in isolation and does not need to be encrypted
- Ensure the pipeline order is: plaintext → generate embedding → encrypt plaintext → store both; decryption must occur before any re-embedding on update

Search &amp; Filtering Compatibility
- Full-text Postgres search (`tsvector`, `ILIKE`) will no longer function on encrypted columns — document this explicitly and confirm that all full-text search paths have already migrated to vector embedding search (which operates on the embedding, not the stored text)
- Any remaining keyword-filter queries against encrypted columns must be replaced with semantic search or moved to application-side filtering post-decryption

---

Implementation Steps

1. Create `lib/encryption.ts` with AES-256-GCM encrypt/decrypt utilities and tests
2. Audit all Drizzle schema files and produce a definitive list of columns to encrypt
3. Write and run a one-time migration script to encrypt all existing plaintext rows in production (encrypt-in-place, column by column, with progress logging and a dry-run mode)
4. Update all Drizzle insert/update call sites to encrypt before write
5. Update all Drizzle select call sites to decrypt after read
6. Update the Open Brain MCP server tools to decrypt thought content before returning to AI clients
7. Update the Krisp webhook ingestion pipeline to encrypt transcript and key-point payloads on arrival
8. Update the email ingestion pipeline (Graph API + Gmail) to encrypt subject, body, and sender fields on arrival
9. Add startup validation that confirms `ENCRYPTION_KEY` is present and valid length
10. Write integration tests that assert: (a) raw database reads return ciphertext, not plaintext; (b) application API responses return correct plaintext; (c) a missing or wrong key causes decryption to throw, not silently return garbage

---

Edge Cases &amp; Considerations

- Null and empty string handling — `null` stays `null`; empty string should encrypt to a non-empty ciphertext (do not special-case as null)
- Partial migration failure — the in-place migration must be idempotent; detect already-encrypted rows by prefix marker (e.g., `enc:v1:`) and skip them
- Session and cache invalidation — if any encrypted field values are cached in Redis, Next.js cache, or React Query, ensure cache keys or cache entries are invalidated after re-encryption during key rotation
- API responses — decryption errors must return a 500 with a generic message, never expose the exception detail or partial plaintext to the client
- Developer local environments — provide a deterministic test key for local development; document that production and staging must use separate, independently rotated keys
- Backup and restore — encrypted backups of Neon remain encrypted at rest; document that a database restore without the corresponding `ENCRYPTION_KEY` value renders all user content unrecoverable; enforce key escrow policy</description>

## Block Editor Behavior

This is the core of the app. It must feel fast and intuitive.

### Text Editing
- Each block is an independently editable region
- Pressing **Enter** at end of a block creates a new `paragraph` block below
- Pressing **Enter** in the middle of a block splits it into two blocks
- Pressing **Backspace** at the start of an empty block deletes it and moves cursor to previous block
- Pressing **Backspace** at the start of a non-empty block merges it with the previous block
- **Arrow Up/Down** at the top/bottom of a block moves cursor to adjacent block

### Slash Commands
- Typing `/` at the start of an empty block (or after a space) opens the slash command menu

## Authentication Flow

1. **Register**: POST email, name, password → hash password with bcrypt → insert user → create default workspace &quot;My Workspace&quot; → create a default &quot;Getting Started&quot; page with sample blocks → return session
2. **Login**: Email + password → verify with bcrypt → issue JWT session
3. **Session**: JWT stored in HTTP-only cookie, includes `{ userId, email, workspaceId }`
4. **Middleware**: Protect all `/api/*` routes (except auth) and `/workspace/*` routes. Redirect unauthenticated users to `/login`.

## What Gets Embedded

Concatenate these fields into a single string before sending to OpenAI:

```
Subject: {subject}
From: {sender}
{body_plain_text truncated to 6000 characters}
```

- Use `body_plain_text` over `body_html` — cleaner signal, no tag noise
- Truncate to stay within the 8192 token limit of `text-embedding-3-small`
- Strip excessive whitespace and newlines before embedding

## Embedding Pipeline

Embeddings are generated **asynchronously** — never on the ingest request path.

```
Email saved to DB
      │
      ▼
Queue job (BullMQ / pg-boss / etc.)
      │
      ▼
Worker pulls emails WHERE embedding IS NULL
      │
      ▼
Calls OpenAI Embeddings API (batch up to 100 at a time)
      │
      ▼
UPDATE emails SET embedding = $1, embedding_generated_at = now()
WHERE id = $2
```

### Worker implementation notes

- **Batch size:** Up to 100 emails per API call (OpenAI supports array input)
- **Retry logic:** Exponential backoff on 429 rate limit responses
- **Backfill:** On first deploy, queue all existing emails with `embedding IS NULL`
- **Frequency:** Process new emails within 60 seconds of insert — near real-time is fine, instant is not required

## New API Route

### `GET /api/emails/search?q=your+query&amp;limit=20`

**Flow:**

1. Receive query string from user
2. Call OpenAI to embed the query (same model, same dimensions)
3. Run cosine similarity search against `emails` table scoped to `tenant_id`
4. Return ranked results

**Query:**

```sql
SELECT
  id, sender, subject, received_at, recipients,
  attachments_metadata,
  LEFT(body_plain_text, 200) AS preview,
  1 - (embedding &lt;=&gt; $2) AS similarity
FROM emails
WHERE
  tenant_id = $1
  AND embedding IS NOT NULL
ORDER BY embedding &lt;=&gt; $2
LIMIT $3;
```

**Response:**

```json
{
  &quot;query&quot;: &quot;invoice payment overdue&quot;,
  &quot;data&quot;: [

      &quot;subject&quot;: &quot;Outstanding Balance Reminder&quot;,
      &quot;received_at&quot;: &quot;2026-02-14T09:00:00Z&quot;,
      &quot;similarity&quot;: 0.91,
      &quot;preview&quot;: &quot;This is a reminder that your account has an unpaid...&quot;

}
```

## Hybrid Search (Recommended)

Pure vector search occasionally misses obvious keyword matches. Run both in parallel and merge:

```typescript
async function hybridSearch(tenantId: string, query: string, limit: number) {
  const [vectorResults, keywordResults] = await Promise.all([
    vectorSearch(tenantId, query, limit),
    keywordSearch(tenantId, query, limit),   // existing ILIKE query
  ]);

  // Merge by id, boost items that appear in both result sets
  return mergeAndRank(vectorResults, keywordResults, limit);
}
```

Simple merge strategy: deduplicate by `id`, assign a combined score where appearing in both lists gets a 1.2x boost. No ML required.

### 1. Inbox List — `GET /inbox`

**Purpose:** Paginated, reverse-chronological list of messages for the authenticated tenant.

**URL Parameters**

| Param | Type | Default | Description |
|---|---|---|---|
| `page` | number | 1 | Page number (1-indexed) |
| `limit` | number | 50 | Rows per page (max 100) |
| `q` | string | — | Full-text search across `sender` and `subject` |
| `after` | ISO date | — | Filter messages received after this date |
| `before` | ISO date | — | Filter messages received before this date |

**SQL (core list query)**

```sql
SELECT
  id, sender, subject, received_at,
  recipients,
  attachments_metadata,
  LEFT(body_plain_text, 200) AS preview
FROM emails
WHERE
  tenant_id = $1
  AND ($2::text IS NULL OR sender ILIKE &apos;%&apos; || $2 || &apos;%&apos; OR subject ILIKE &apos;%&apos; || $2 || &apos;%&apos;)
  AND ($3::timestamptz IS NULL OR received_at &gt;= $3)
  AND ($4::timestamptz IS NULL OR received_at &lt;= $4)
ORDER BY received_at DESC
LIMIT $5 OFFSET $6;
```

This query uses `idx_emails_tenant_received`. For full-text search at scale, promote `subject` and `sender` to a `tsvector` generated column with a GIN index.

## Performance Considerations

- The composite index `idx_emails_tenant_received (tenant_id, received_at)` is the primary sort index for the inbox query — no additional indexes needed for baseline performance.
- For tenants with &gt; 100 k messages, add keyset pagination (`WHERE received_at &lt; $cursor`) instead of `OFFSET` to avoid deep-page slowdowns.
- Cache the inbox list response in Next.js with `revalidate: 30` (ISR) if the data is not real-time critical, or use `no-store` for live inbox behavior.
- `body_html` and `body_plain_text` are excluded from the list query; only fetched on the detail page.

## How Gmail &quot;Webhooks&quot; Actually Work

Gmail doesn&apos;t support direct webhooks, but Google provides an official push notification system:

1. **Gmail Watch API** — tells Gmail to monitor an inbox and publish notifications to a Google Pub/Sub topic when new messages arrive
2. **Google Pub/Sub** — receives those notifications and immediately pushes them to your HTTPS endpoint
3. **Your backend** — receives the Pub/Sub push, uses the Gmail API to fetch the actual email content, and ingests it

This is the production-grade approach used by enterprise Gmail integrations. Alternatively, for tenants who prefer a no-code setup, **Google Apps Script** can poll Gmail and POST to your webhook on a time trigger — this is simpler but introduces up to a 1-minute delay.

---

## Webhook Endpoint

**Primary endpoint (Pub/Sub push):**
```
POST /webhooks/email/gmail/:tenantId
```

**Authentication:** Pub/Sub push requests include a Bearer token in the `Authorization` header issued by Google. Verify this token against Google&apos;s token info endpoint (`https://oauth2.googleapis.com/tokeninfo`) to confirm it&apos;s a legitimate Pub/Sub push and that the `email` claim matches your registered service account. Additionally, validate a `?token=&lt;secret&gt;` query parameter appended to the subscription push URL during setup as a secondary layer.

**What Pub/Sub delivers:** The push payload is a Pub/Sub message envelope — not the email itself. The `data` field is base64-encoded and contains only the Gmail `historyId` and the user&apos;s email address. Your backend must then call the Gmail API to retrieve the actual message.

**Ingestion flow:**

```
Pub/Sub Push → Your endpoint → Decode historyId → Gmail API (history.list) 
→ Gmail API (messages.get) → Parse &amp; store email
```

**Parse and extract from Gmail API response:**
- `from` header → sender address
- `to`, `cc`, `bcc` headers → recipients
- `subject` header → subject
- `body.data` (text/plain part) → plain text body
- `body.data` (text/html part) → HTML body
- `internalDate` → received timestamp
- `id` → Gmail message ID (used for deduplication)
- `threadId` → conversation thread reference
- `payload.parts` where `filename` is non-empty → attachment metadata (name, MIME type, size, attachment ID)

**HTTP responses:**
- `200` — success or duplicate (Pub/Sub will not retry on 200)
- `400` — payload malformed or tenant resolution failed
- `401` — auth token invalid
- `500` — internal error (Pub/Sub **will** retry, so only use this when a retry is appropriate)

&gt; **Critical:** Return `200` for duplicates and known-bad messages you&apos;ve already handled. Returning non-2xx causes Pub/Sub to retry with exponential backoff for up to 7 days.

## Database

```sql
CREATE TABLE emails (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id         UUID NOT NULL REFERENCES tenants(id),
  gmail_message_id  TEXT NOT NULL,
  thread_id         TEXT,
  sender            TEXT NOT NULL,
  recipients        JSONB NOT NULL DEFAULT &apos;[]&apos;,
  subject           TEXT,
  body_plain        TEXT,
  body_html         TEXT,
  received_at       TIMESTAMPTZ NOT NULL,
  attachments       JSONB NOT NULL DEFAULT &apos;[]&apos;,
  labels            JSONB NOT NULL DEFAULT &apos;[]&apos;,
  raw_payload       JSONB,
  ingested_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),

CREATE INDEX idx_emails_tenant_id ON emails (tenant_id);
CREATE INDEX idx_emails_received_at ON emails (tenant_id, received_at DESC);
CREATE INDEX idx_emails_sender ON emails (tenant_id, sender);
```

**Notes:**
- `body_plain` and `body_html` are `TEXT` (unbounded) — avoid `VARCHAR(n)` for email bodies
- `recipients`, `attachments`, and `labels` use `JSONB` to accommodate variable-length arrays without separate join tables
- `raw_payload` stores the full Gmail API `Message` object for debugging; consider stripping body content from this field to avoid double-storing large content
- All queries must include `WHERE tenant_id = $1` as the first filter condition

## Gmail OAuth &amp; Watch Setup (Per Tenant)

Each tenant must complete a one-time OAuth authorization before their inbox can be monitored.

**Required Gmail API scopes:**
- `https://www.googleapis.com/auth/gmail.readonly` — to fetch message content
- `https://www.googleapis.com/auth/gmail.modify` — only if you need to mark messages read or apply labels

**Watch registration** (call this after OAuth completes, and refresh every 7 days — Gmail watch subscriptions expire):

```json
POST https://gmail.googleapis.com/gmail/v1/users/me/watch
{
  &quot;topicName&quot;: &quot;projects/YOUR_PROJECT/topics/gmail-inbound&quot;,
  &quot;labelIds&quot;: [&quot;INBOX&quot;],
  &quot;labelFilterBehavior&quot;: &quot;INCLUDE&quot;
}
```

Store the returned `historyId` as the tenant&apos;s baseline — you&apos;ll use this to call `history.list` incrementally so you don&apos;t re-process old messages.

**Watch renewal:** Store the `expiration` timestamp from the watch response and set up a daily job to renew watches expiring within 24 hours.

## Multi-Tenant Support

Tenant resolution uses the `:tenantId` URL path parameter combined with a per-tenant API key stored in your database.

**Tenant record should include:**
- `gmail_oauth_tokens` (encrypted) — access token, refresh token, expiry
- `gmail_watch_history_id` — the last processed `historyId` for incremental fetching
- `gmail_watch_expiration` — when the Pub/Sub watch subscription expires
- `webhook_api_key` — the secret appended to the Pub/Sub push URL for secondary auth validation

**Isolation guarantees:**
- All Gmail API calls use the OAuth tokens belonging to the resolved tenant only
- The `historyId` cursor is stored per-tenant — no shared state
- All database writes and reads are scoped to `tenant_id`
- The Pub/Sub topic is shared infrastructure, but tenant identity is resolved from the push URL before any data access occurs

## Apps Script Alternative (No-Code Setup)

For tenants who can&apos;t complete OAuth flows or prefer a simpler setup, provide a Google Apps Script template they can deploy themselves:

```javascript
function checkNewEmails() {
  const webhookUrl = &quot;https://yourapp.com/webhooks/email/gmail/TENANT_ID&quot;;
  const apiKey = &quot;TENANT_API_KEY&quot;;
  const lastRun = PropertiesService.getScriptProperties()
                    .getProperty(&quot;lastRun&quot;) || &quot;2000/01/01&quot;;
  
  const threads = GmailApp.search(`after:${lastRun} in:inbox`);
  
  threads.forEach(thread =&gt; {
    thread.getMessages().forEach(msg =&gt; {
      const payload = {
        messageId: msg.getId(),
        sender: msg.getFrom(),
        recipients: msg.getTo(),
        subject: msg.getSubject(),
        bodyPlain: msg.getPlainBody(),
        bodyHtml: msg.getBody(),
        receivedAt: msg.getDate().toISOString()
      };
      
      UrlFetchApp.fetch(webhookUrl, {
        method: &quot;POST&quot;,
        headers: { &quot;Content-Type&quot;: &quot;application/json&quot;, &quot;X-API-Key&quot;: apiKey },
        payload: JSON.stringify(payload),
        muteHttpExceptions: true
      });
    });
  });
  
  PropertiesService.getScriptProperties()
    .setProperty(&quot;lastRun&quot;, new Date().toLocaleDateString(&quot;en-US&quot;));
}
```

Tenants set a time-based trigger (every 1 minute) in Apps Script. This approach doesn&apos;t require Pub/Sub or OAuth on your backend — the payload arrives pre-parsed, so your endpoint can skip the Gmail API fetch step and ingest directly. The tradeoff is up to a 1-minute delay and reliance on the tenant keeping their Apps Script project active.

## Edge Cases

**Duplicate delivery:** The `UNIQUE (tenant_id, gmail_message_id)` constraint is the authoritative deduplication guard. Use an `INSERT ... ON CONFLICT DO NOTHING` pattern and return `200` either way so Pub/Sub doesn&apos;t retry.

**Missing fields:** Subject, plain body, and HTML body should all be nullable at the application layer. Emails with no subject (common for automated senders) or no plain text body (HTML-only emails) are valid and should be stored as-is.

**Large bodies:** Gmail messages can theoretically be very large. Impose a configurable size cap (e.g., 5MB) on `body_html` and `body_plain` before storage, truncating with a metadata flag (`body_truncated: true`) so downstream consumers know the content is incomplete rather than silently missing.

**Expired watch subscriptions:** If a tenant&apos;s Gmail watch expires and isn&apos;t renewed, Pub/Sub stops receiving notifications silently — no error is thrown. Show an &quot;Integration Inactive — watch expired&quot; warning in the UI based on the stored expiration timestamp, and send the tenant an email alert 48 hours before expiry.

**OAuth token refresh:** Access tokens expire every 60 minutes. Always attempt a refresh before making Gmail API calls and update the stored tokens. If a refresh token is revoked (tenant disconnects the app from their Google Account), mark the integration as inactive and alert the tenant.

**historyId gaps:** If your server is down when Pub/Sub delivers a notification, you may miss messages. On reconnection, call `history.list` starting from the last stored `historyId` to backfill any gaps. Cap backfill lookback to 7 days, which is Gmail&apos;s history retention window.

**Attachments:** Only store attachment metadata (filename, MIME type, size, Gmail attachment ID) in the database by default. Actual attachment content should be fetched on-demand via the Gmail API using the stored attachment ID, rather than eagerly downloading and storing binary content at ingestion time.</description>

## 1. Executive Summary

This specification defines a complete subscription billing service for a SaaS application, built on Next.js (App Router) and the Stripe SDK. It covers the full billing lifecycle: plan selection, checkout, recurring payment processing, invoice management, payment method updates, plan upgrades/downgrades, cancellations, and administrative oversight.

The system is designed around three core principles drawn from industry best practices:

- **Stripe as the billing engine, your database as the source of truth.** Stripe handles payment processing, invoicing, dunning, and card management. Your application database stores customer entitlements, subscription status, and access rules.

- **Webhook-driven state management.** Subscription state changes are confirmed via Stripe webhooks, not by trusting checkout success URLs. This ensures reliability across network failures, 3D Secure flows, and async payment methods.

- **Self-service first.** Customers can select plans, update payment methods, view invoices, and cancel subscriptions without contacting support. Administrators have a dedicated dashboard for oversight and intervention.

### 2.2 High-Level Data Flow

The system follows this flow for a new subscription:

1. User selects a plan on the Pricing Page.
2. Next.js Server Action creates a Stripe Checkout Session.
3. User is redirected to Stripe-hosted Checkout to enter payment details.
4. Stripe processes payment, creates the Subscription, Invoice, and PaymentIntent objects.
5. Stripe sends webhook events to the application&apos;s webhook endpoint.
6. Webhook handler verifies the signature, processes events, and updates the application database.
7. User is redirected to the success page; access is granted based on database state (not the redirect).

## 4. Database Schema

### 4.1 Prisma Schema

The database stores the application&apos;s view of customer and subscription state. Stripe is the billing engine, but your database defines entitlements and access rules. The following Prisma models capture the essential data.

#### 4.1.1 User Model

#### 4.1.2 Subscription Model

```prisma
model Subscription {
  id                      String             @id @default(cuid())
  userId                  String             @unique
  user                    User               @relation(fields: [userId], references: [id])
  stripeSubscriptionId    String             @unique
  stripePriceId           String
  stripeCurrentPeriodEnd  DateTime
  status                  SubscriptionStatus
  cancelAtPeriodEnd       Boolean            @default(false)
  createdAt               DateTime           @default(now())
  updatedAt               DateTime           @updatedAt
```

#### 4.1.3 Enums

```prisma

  ACTIVE
  TRIALING
  PAST_DUE
  CANCELED
  UNPAID
  INCOMPLETE

enum Role {
  USER
  ADMIN
}
```

### 4.2 Key Design Decisions

- **Store Stripe IDs.** Always persist `stripeCustomerId` and `stripeSubscriptionId`. Without them, reconciling billing issues becomes extremely difficult.
- **Single subscription per user.** This spec assumes one active subscription per user. For multi-subscription models, change the relation to one-to-many.
- **`stripeCurrentPeriodEnd` as the access gate.** When checking access, compare the current time against this timestamp (with a 1–2 day buffer for payment retries).

### 6.1 Pricing Page (`/pricing`)

The pricing page is the most conversion-critical page in the application. Industry research consistently shows that three to four tiers with a highlighted recommended plan produces the highest conversion rates.

#### 6.1.1 Layout and Structure

- Display plans in a horizontal card layout (vertical stack on mobile)
- Include a monthly/annual toggle switch, with the annual discount prominently shown
- Visually highlight the recommended plan with a distinct border, badge, or background tint
- Each plan card shows: plan name, price, billing interval, a concise feature list, and a primary CTA button
- Add a feature comparison table below the cards for users who want granular detail

#### 6.1.2 Conversion Best Practices

- **Anchoring:** Display the highest-priced plan first (left to right) so mid-tier plans feel like good value.
- **Social proof:** Include customer counts, testimonials, or trust badges near the pricing cards.
- **FAQ section:** Address common concerns (refund policy, cancellation, hidden fees) directly on the page.
- **Transparent pricing:** No hidden fees. Show exactly what users will be charged. This is the single most impactful trust signal.
- **Free trial option:** If offering a trial, reduce friction by not requiring a credit card upfront (or clearly communicate that it is required).

#### 6.1.3 Technical Implementation

The pricing page is a Server Component that fetches plan data from environment configuration (or from Stripe via the Products API for dynamic pricing). The Subscribe button triggers a Server Action that creates a Stripe Checkout Session:

```typescript
// app/pricing/actions.ts
&apos;use server&apos;

import { stripe } from &apos;@/lib/stripe&apos;;
import { redirect } from &apos;next/navigation&apos;;

export async function createCheckoutSession(priceId: string) {
  const user = await getCurrentUser();

  // Create or retrieve the Stripe customer
  let stripeCustomerId = user.stripeCustomerId;
  if (!stripeCustomerId) {
    const customer = await stripe.customers.create({
      email: user.email,
      metadata: { userId: user.id },
    });
    stripeCustomerId = customer.id;
    await db.user.update({
      where: { id: user.id },
      data: { stripeCustomerId },
    });
  }

  const session = await stripe.checkout.sessions.create({
    customer: stripeCustomerId,
    mode: &apos;subscription&apos;,
    payment_method_types: [&apos;card&apos;],
    line_items: [{ price: priceId, quantity: 1 }],
    success_url: `${process.env.NEXT_PUBLIC_APP_URL}/checkout/success?session_id={CHECKOUT_SESSION_ID}`,
    cancel_url: `${process.env.NEXT_PUBLIC_APP_URL}/checkout/cancel`,
    subscription_data: {
      metadata: { userId: user.id },
    },
  });

  redirect(session.url!);
}
```

## 7. Webhook Handler

### 7.1 Endpoint Setup

The webhook endpoint is a Next.js Route Handler at `/api/webhooks/stripe/route.ts`. It must use the raw request body (not parsed JSON) for signature verification. This is critical for security.

```typescript
// app/api/webhooks/stripe/route.ts
import { NextResponse } from &apos;next/server&apos;;
import { stripe } from &apos;@/lib/stripe&apos;;
import { db } from &apos;@/lib/db&apos;;

export async function POST(req: Request) {
  const body = await req.text();
  const sig = req.headers.get(&apos;stripe-signature&apos;)!;

  let event;
  try {
    event = stripe.webhooks.constructEvent(
      body,
      sig,
      process.env.STRIPE_WEBHOOK_SECRET!
    );
  } catch (err) {
    console.error(&apos;Webhook signature verification failed:&apos;, err);
    return NextResponse.json({ error: &apos;Invalid signature&apos; }, { status: 400 });
  }

  try {
    switch (event.type) {
      case &apos;checkout.session.completed&apos;:
        await handleCheckoutCompleted(event.data.object);
        break;
      case &apos;invoice.paid&apos;:
        await handleInvoicePaid(event.data.object);
        break;
      case &apos;invoice.payment_failed&apos;:
        await handlePaymentFailed(event.data.object);
        break;
      case &apos;customer.subscription.updated&apos;:
        await handleSubscriptionUpdated(event.data.object);
        break;
      case &apos;customer.subscription.deleted&apos;:
        await handleSubscriptionDeleted(event.data.object);
        break;
      default:
        console.log(`Unhandled event type: ${event.type}`);
    }
  } catch (err) {
    console.error(`Error processing ${event.type}:`, err);
    // Still return 200 to prevent Stripe retries for application errors
    // Log the error for investigation
  }

  return NextResponse.json({ received: true });
}
```

### 7.2 Event Processing Logic

Each webhook event triggers specific database updates. The handler must be idempotent because Stripe retries failed deliveries for up to 3 days with exponential backoff.

#### 7.2.1 `checkout.session.completed`

```typescript
async function handleCheckoutCompleted(session: Stripe.Checkout.Session) {
  const subscription = await stripe.subscriptions.retrieve(
    session.subscription as string
  );

  await db.subscription.upsert({
    where: { userId: subscription.metadata.userId },
    create: {
      userId: subscription.metadata.userId,
      stripeSubscriptionId: subscription.id,
      stripePriceId: subscription.items.data[0].price.id,
      stripeCurrentPeriodEnd: new Date(subscription.current_period_end * 1000),
      status: &apos;ACTIVE&apos;,
    },
    update: {
      stripeSubscriptionId: subscription.id,
      stripePriceId: subscription.items.data[0].price.id,
      stripeCurrentPeriodEnd: new Date(subscription.current_period_end * 1000),
      status: &apos;ACTIVE&apos;,
    },
  });
}
```

- Extract the Stripe customer ID and subscription ID from the session object
- Create or update the Subscription record in the database
- Link the Stripe customer ID to the User record if not already linked
- Set subscription status to `ACTIVE` (or `TRIALING` if a trial was configured)

#### 7.2.2 `invoice.paid`

- Look up the subscription by `stripeSubscriptionId`
- Update `stripeCurrentPeriodEnd` to the new period end date
- Ensure status is `ACTIVE`
- This event confirms continued access on each renewal cycle

#### 7.2.3 `invoice.payment_failed`

- Update subscription status to `PAST_DUE`
- Trigger a notification to the customer (email or in-app) prompting them to update their payment method
- Stripe will automatically retry the payment based on your configured retry schedule

#### 7.2.4 `customer.subscription.updated`

- Update the local subscription record with the new price ID, status, and period end
- This handles plan changes (upgrades/downgrades), proration adjustments, and trial-to-active transitions

#### 7.2.5 `customer.subscription.deleted`

- Set subscription status to `CANCELED`
- Revoke access to premium features
- Optionally retain the subscription record for historical reference rather than deleting it

### 7.3 Idempotency

Stripe may deliver the same event multiple times. Implement idempotency by storing processed event IDs (the `event.id` field) in a database table or cache. Before processing an event, check if it has already been handled. A simple lookup table keyed by event ID with a TTL of 72 hours is sufficient.

```typescript
// Check idempotency before processing
const existing = await db.webhookEvent.findUnique({
  where: { stripeEventId: event.id },
});
if (existing) {
  return NextResponse.json({ received: true });
}

// Process event...

// Mark as processed
await db.webhookEvent.create({
  data: { stripeEventId: event.id, type: event.type },
});
```

### 7.4 Error Handling

- Always return a 200 status code quickly, even if downstream processing fails (queue events for async processing if needed)
- Log webhook verification failures separately for debugging
- In live mode, Stripe retries webhooks for up to 3 days if it doesn&apos;t receive a 2xx response
- Monitor webhook health: track the last received event type and timestamp to detect delivery gaps

## 9. Access Control and Entitlements

### 9.1 Middleware-Based Route Protection

Use Next.js middleware to protect routes based on authentication and subscription status. The middleware checks the user&apos;s session and subscription state before rendering protected pages.

```typescript
// middleware.ts
import { NextRequest, NextResponse } from &apos;next/server&apos;;
import { getToken } from &apos;next-auth/jwt&apos;;

export async function middleware(request: NextRequest) {
  const token = await getToken({ req: request });
  const { pathname } = request.nextUrl;

  // 1. Unauthenticated users → login
  if (!token &amp;&amp; pathname.startsWith(&apos;/billing&apos;)) {
    return NextResponse.redirect(new URL(&apos;/login&apos;, request.url));
  }

  // 2. Admin routes → verify ADMIN role
  if (pathname.startsWith(&apos;/admin&apos;) &amp;&amp; token?.role !== &apos;ADMIN&apos;) {
    return NextResponse.redirect(new URL(&apos;/&apos;, request.url));
  }

  // 3. Premium routes → verify active subscription
  if (pathname.startsWith(&apos;/app&apos;) &amp;&amp; !token?.hasActiveSubscription) {
    return NextResponse.redirect(new URL(&apos;/pricing&apos;, request.url));
  }

  return NextResponse.next();
}

export const config = {
  matcher: [&apos;/billing/:path*&apos;, &apos;/admin/:path*&apos;, &apos;/app/:path*&apos;],
};
```

### 9.2 Feature Gating

Implement a utility function that checks subscription status and plan tier to gate features:

```typescript
// lib/access.ts
import { PLAN_FEATURES } from &apos;./plans&apos;;

export function hasAccess(
  user: { subscription: Subscription | null },
  feature: string
): boolean {
  if (!user.subscription) return false;

  const { status, stripeCurrentPeriodEnd, stripePriceId } = user.subscription;

  const isActive =
    [&apos;ACTIVE&apos;, &apos;TRIALING&apos;].includes(status) &amp;&amp;
    new Date(stripeCurrentPeriodEnd) &gt; new Date();

  if (!isActive) return false;

  return PLAN_FEATURES[stripePriceId]?.includes(feature) ?? false;
}
```

### 9.3 Grace Period

When a payment fails, don&apos;t immediately revoke access. Allow a grace period (typically 3–7 days) aligned with Stripe&apos;s automatic retry schedule. During this period, show a banner prompting the user to update their payment method. Only revoke access after the subscription transitions to `UNPAID` or `CANCELED`.

```typescript
// lib/access.ts
const GRACE_PERIOD_DAYS = 3;

export function hasActiveAccess(subscription: Subscription): boolean {
  if ([&apos;ACTIVE&apos;, &apos;TRIALING&apos;].includes(subscription.status)) {
    return true;
  }

  if (subscription.status === &apos;PAST_DUE&apos;) {
    const gracePeriodEnd = new Date(subscription.stripeCurrentPeriodEnd);
    gracePeriodEnd.setDate(gracePeriodEnd.getDate() + GRACE_PERIOD_DAYS);
    return new Date() &lt; gracePeriodEnd;
  }

  return false;
}
```

## 11. Testing Strategy

### 11.1 Stripe Test Mode

|-------------|----------|

| `4000 0000 0000 9995` | Payment declined (insufficient funds) |
| `4000 0000 0000 0341` | Attach succeeds, but first charge fails |

### 11.2 Stripe CLI for Webhook Testing

The CLI prints a webhook signing secret (`whsec_*`) for local testing. Set this as `STRIPE_WEBHOOK_SECRET` in your `.env.local` file. The development server and webhook forwarding must run concurrently in separate terminals.

### 11.3 Full Lifecycle Test Checklist

Before shipping, validate every stage of the subscription lifecycle:

1. **New subscription:** User selects plan, completes checkout, webhook fires, database updates, access granted.
2. **Successful renewal:** Subscription renews, `invoice.paid` fires, period end updated, access continues.
3. **Failed payment:** Use a declining card, verify `invoice.payment_failed` fires, status becomes `PAST_DUE`, user sees payment update banner.
4. **Plan upgrade:** User upgrades, proration applied, `subscription.updated` fires, new features available.
5. **Plan downgrade:** User downgrades, change takes effect at period end (or immediately with proration).
6. **Cancellation (at period end):** User cancels, access continues until period end, then revoked.
7. **Cancellation (immediate):** Admin cancels immediately, access revoked, `subscription.deleted` fires.
8. **Reactivation:** User reactivates before period end, `cancel_at_period_end` removed.
9. **Payment method update:** User updates card via Customer Portal, new payment method appears.
10. **3D Secure flow:** Use the 3DS test card, verify authentication challenge completes properly.

## 12. Security Considerations

- **Never trust the success URL for access granting.** The checkout success page is cosmetic. Always gate access based on webhook-confirmed subscription state in your database.

- **Verify webhook signatures.** Use `stripe.webhooks.constructEvent` with the webhook signing secret. Never process unverified events. Anyone can POST to your webhook URL.

# Web Clipper MVP — Spec

## 1. What This Is

A Chrome extension that clips the readable content of any web page and a React viewer to read saved clips. Uses Mozilla&apos;s Readability to extract clean article content, DOMPurify to sanitize it, and IndexedDB for local storage.

**MVP scope:** Clip a page → save it → read it later. That&apos;s it.

---

## 2. Architecture

```
Chrome Extension                          React Viewer (new tab)
┌──────────────────────┐                 ┌──────────────────────┐
│  Popup: [Clip] btn   │                 │  Clip List → Reader  │
│          ↓           │                 │          ↑           │
│  Content Script      │                 │          │           │
│  • clone DOM         │   IndexedDB     │          │           │
│  • Readability parse │──── clips ──────│  reads from same DB  │
│  • DOMPurify clean   │   store         │                      │
│  • save to IDB       │                 │  html-react-parser   │
└──────────────────────┘                 └──────────────────────┘
```

No backend. No sync. Everything lives in the browser&apos;s IndexedDB.

---

## 3. Data Model

```typescript
interface Clip {
  id: string;            // crypto.randomUUID()
  url: string;           // original page URL
  title: string;         // Readability title
  byline?: string;       // author if detected
  siteName?: string;     // e.g. &quot;The Verge&quot;
  excerpt: string;       // first ~200 chars, plain text
  content: string;       // sanitized HTML from Readability
  textContent: string;   // plain text version (for search)
  clippedAt: string;     // ISO 8601
  tags: string[];        // user-assigned
  readingTimeMin: number; // Math.ceil(wordCount / 238)
}
```

**IndexedDB:** Database `web-clipper`, object store `clips` keyed on `id`, indexed on `clippedAt` and `tags` (multiEntry).

---

## 4. Chrome Extension

### 4.1 Manifest v3

```jsonc
{
  &quot;manifest_version&quot;: 3,
  &quot;name&quot;: &quot;Web Clipper&quot;,
  &quot;version&quot;: &quot;0.1.0&quot;,
  &quot;permissions&quot;: [&quot;activeTab&quot;, &quot;scripting&quot;],
  &quot;action&quot;: { &quot;default_popup&quot;: &quot;popup.html&quot; },
  &quot;background&quot;: { &quot;service_worker&quot;: &quot;background.js&quot;, &quot;type&quot;: &quot;module&quot; }
}
```

Only `activeTab` + `scripting` — no broad host permissions needed since we inject on demand.

### 4.2 Clipping Pipeline (Content Script)

Injected via `chrome.scripting.executeScript` when the user clicks Clip.

1. `document.cloneNode(true)` — don&apos;t mutate the live page.
2. `new Readability(clone).parse()` → `{ title, byline, content, textContent, excerpt, siteName }`.
3. `DOMPurify.sanitize(content, config)` — strict allowlist (see §6).
4. Compute `readingTimeMin` from word count.
5. Build `Clip` object, `chrome.runtime.sendMessage()` to background.

If Readability returns `null`, fall back to `document.title` + truncated `body.innerText`. Flag it in the popup as a partial clip.

### 4.3 Background Service Worker

Receives the clip, writes it to IndexedDB, sends a response back to the popup with success/failure.

### 4.4 Popup UI

Three states:

- **Ready** — Shows a &quot;Clip This Page&quot; button. Optional tag input field (comma-separated).
- **Saving** — Spinner / &quot;Clipping…&quot; text.
- **Done** — &quot;Saved!&quot; with a link to open the viewer.

That&apos;s the entire popup. Keep it minimal.

---

|---|---|
