  <project_name>Krisp — Open Brain Productivity Hub</project_name>

  <overview>
    Krisp is a comprehensive, database-backed productivity and AI knowledge hub that combines Kanban task management, multi-source email aggregation, meeting intelligence (via Krisp webhook transcripts), and an &quot;Open Brain&quot; persistent memory system — all unified under a single Next.js application with a Neon PostgreSQL database.

The project solves the &quot;memory problem&quot; in AI workflows: every time you open a new chat window or switch AI tools, your accumulated context resets to zero. Krisp addresses this by providing (1) a centralized productivity dashboard for managing tasks, emails, and meeting transcripts with semantic search, and (2) an Open Brain architecture — a database-backed, MCP-connected knowledge system where any AI tool (Claude, ChatGPT, Cursor, etc.) can query your accumulated context through a single open protocol.

Key goals:
- Eliminate context silos across AI tools by providing persistent, semantically searchable memory accessible via MCP
- Aggregate meetings, emails, and tasks into a single productivity surface with AI-powered search
- Enable sub-second thought capture (via Slack or any MCP client) with automatic embedding, classification, and metadata extraction
- Provide compounding knowledge advantage: every captured thought makes every future AI interaction richer
- Maintain full user ownership of data — no SaaS middlemen, single Postgres database, ~$0.10–$0.30/month operational cost
- Support the agent web: make all knowledge queryable by meaning (vector embeddings), not just keywords, through standard protocols any autonomous agent can speak
  </overview>

Requirements:

Webhook Endpoint
- Create a new POST webhook endpoint (e.g., /webhooks/email/microsoft365) that receives email payloads triggered by Power Automate when a new email arrives in Exchange
- Authenticate incoming requests (e.g., via a shared secret or API key header) to prevent unauthorized submissions

Database
- Design and create a new emails table to store ingested email records with columns for: tenant ID, message ID (for deduplication), sender, recipients, subject, body content, received timestamp, raw payload, and created/updated timestamps
- Add a unique constraint on tenant ID + message ID to prevent duplicate ingestion
- Ensure all queries are scoped by tenant ID to enforce multi-tenant data isolation

Multi-Tenant Support
- Resolve the tenant from the incoming webhook request (e.g., via a tenant-specific API key, a URL path parameter such as /webhooks/email/microsoft365/:tenantId, or a JWT claim)
- Ensure no cross-tenant data leakage at the query, storage, and response layer

Admin Integrations Page
- Create a new admin page at /admin/integrations
- Include a clearly titled &quot;Microsoft 365 / Exchange Email&quot; section with step-by-step instructions for setting up a Power Automate flow, covering:

  - Action: HTTP POST to the tenant&apos;s unique webhook URL
  - Required headers (Content-Type, auth key)
  - A mapping guide showing which Power Automate dynamic fields (e.g., From, Subject, Body) map to which webhook payload fields
  - How to retrieve the tenant&apos;s API key or webhook URL from within the admin UI

Edge Cases to Consider
- Handle duplicate deliveries gracefully using message ID deduplication
- Accommodate emails with missing or null optional fields (e.g., no subject, no plain-text body)
- Handle large email bodies without exceeding payload or database column size limits
- Validate that the tenant exists and is active before persisting any data
- Log ingestion failures with enough context for debugging without storing sensitive content

Prerequisites
- Existing multi-tenant architecture and tenant resolution mechanism in place
- Admin authentication and role-based access controlling the /admin/integrations route</description>

    <guideline>Use Zod schemas for runtime validation on all API inputs (lib/validators/schemas.ts)</guideline>
    <guideline>Follow Next.js App Router conventions with route groups: (app) for protected routes, (auth) for public auth routes</guideline>
    <guideline>Mark client-side components with &apos;use client&apos; directive; keep server components as default</guideline>
    <guideline>Use TanStack React Query via custom hooks (lib/hooks/) for all client-side data fetching and caching</guideline>
    <guideline>Apply TypeScript strict mode with no implicit any — all types defined in types/ directory</guideline>
    <guideline>Store JSONB columns for flexible/evolving data structures (speakers, participants, rawPayload, attachments)</guideline>
    <guideline>Use pgvector(1536) for all embedding columns to enable semantic search</guideline>

    <guideline>Prefer specific file staging over &apos;git add -A&apos; to avoid committing .env or credential files</guideline>
    <guideline>Delete .next directory when encountering stale TypeScript type errors after schema changes</guideline>
    <guideline>All webhook endpoints must validate payload authenticity before processing</guideline>
